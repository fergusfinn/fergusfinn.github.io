Disaggregated inference splits serving across two machines. Prefill — running the prompt through the model to build its KV cache — is compute-bound and bursty; decode — generating tokens off that cache — is memory-bound and steady. On one GPU they contend; split apart, each can be sized for what it does. The catch is the seam: the KV cache built on the prefill GPU must reach the decode GPU before generation starts, and for a long prompt that's a lot of bytes that have to move fast enough not to erase the gain.

This post follows one such transfer all the way down — a single request's KV cache, prefill node to decode node, through every layer that touches it. vLLM decides it should happen; NIXL abstracts it; UCCL carries it; on a Slingshot machine the CXI provider and libfabric turn it into a hardware command, down to the single store that pokes the NIC. Each layer has one job. We name the job, then drop through its floor to the next.

*Pinned to `doublewordai/vllm@e2271c3`, `ai-dynamo/nixl@f4b10ff`, `doublewordai/uccl@c728050`, and below libfabric to `ofiwg/libfabric@24d6319`, `shs-cassini-headers@06f69b0`, `shs-libcxi@a962a9`. We follow the unrestricted DMA read — the path a KV read takes.*

## vLLM: deciding to move

vLLM sits at the top because it's the only layer that understands what's being moved. Below it, everything is buffers and byte ranges; here a transfer is still a *request*, its KV cache made of *blocks*, on one of two machines with *roles* — prefill or decode. vLLM's job is orchestration, not movement: decide a request needs remote KV, line up the memory, drive the lifecycle. It is configured with a KV connector — here [`NixlConnector`](https://github.com/doublewordai/vllm/blob/e2271c323ed3da1dd3827cc84b142b8b621e61b5/vllm/config/kv_transfer.py#L23-L125) — and never copies a byte itself.

Get one thing straight first — who moves the data, because the obvious answer is wrong. When a prefill request finishes, vLLM sends nothing. It holds onto the blocks, delaying the free that would normally reclaim them, and [returns a bundle of coordinates](https://github.com/doublewordai/vllm/blob/e2271c323ed3da1dd3827cc84b142b8b621e61b5/vllm/distributed/kv_transfer/kv_connector/v1/nixl/scheduler.py#L580-L650): which blocks, which engine, which host and port. That bundle reaches the decode node as ordinary request metadata. Decode reads it, allocates local blocks to receive the data, and *pulls* the cache across with a remote read. Prefill publishes; decode fetches. The cache moves because someone downstream asked, not because someone upstream pushed.

Mechanically, decode stores those coordinates as [`kv_transfer_params`](https://github.com/doublewordai/vllm/blob/e2271c323ed3da1dd3827cc84b142b8b621e61b5/vllm/v1/request.py#L101-L117) on the request, and the [scheduler](https://github.com/doublewordai/vllm/blob/e2271c323ed3da1dd3827cc84b142b8b621e61b5/vllm/v1/core/sched/scheduler.py#L597-L650) asks the connector how many tokens it can supply from remote KV, allocates destination blocks, and — if the load is async — [parks the request in `WAITING_FOR_REMOTE_KVS`](https://github.com/doublewordai/vllm/blob/e2271c323ed3da1dd3827cc84b142b8b621e61b5/vllm/v1/core/sched/scheduler.py#L763-L770) instead of scheduling it for compute. It sits there, consuming no model time, until the pull completes. On the next worker step, [`ActiveKVConnector.pre_forward`](https://github.com/doublewordai/vllm/blob/e2271c323ed3da1dd3827cc84b142b8b621e61b5/vllm/v1/worker/gpu/kv_connector.py#L62-L76) hands the connector its metadata and calls [`start_load_kv`](https://github.com/doublewordai/vllm/blob/e2271c323ed3da1dd3827cc84b142b8b621e61b5/vllm/distributed/kv_transfer/kv_connector/v1/nixl/worker.py#L1933-L1968) — the call that turns coordinates into a transfer. It handshakes with the remote agent if needed, works out which local and remote blocks line up, and [posts a NIXL `READ`](https://github.com/doublewordai/vllm/blob/e2271c323ed3da1dd3827cc84b142b8b621e61b5/vllm/distributed/kv_transfer/kv_connector/v1/nixl/worker.py#L2217-L2233). Past that, vLLM has stopped thinking about bytes; it has handed a transfer to NIXL and only checks back to see whether it finished.

## NIXL: one API over every fabric

NIXL's job is to make every transport look the same. vLLM posts a read without caring whether the bytes cross RDMA, TCP, NVMe, or — here — UCCL; NIXL gives it one read/write API and dispatches to whichever backend is configured. It knows nothing of tokens or requests. Its unit is the *descriptor*: a handle into a registered region of memory, a (location, offset, length) the backend can move.

So NIXL must be told about memory before it can move any. At registration, vLLM [walks its KV cache tensors](https://github.com/doublewordai/vllm/blob/e2271c323ed3da1dd3827cc84b142b8b621e61b5/vllm/distributed/kv_transfer/kv_connector/v1/nixl/worker.py#L788-L1013), turns each K and V region into a descriptor, and registers them with the backend. It also builds [`NixlAgentMetadata`](https://github.com/doublewordai/vllm/blob/e2271c323ed3da1dd3827cc84b142b8b621e61b5/vllm/distributed/kv_transfer/kv_connector/v1/nixl/metadata.py#L42-L72), a blob describing the local agent's layout, which the two sides exchange out of band. The pull depends on this: before decode can read from prefill, it has to know prefill's layout, and that knowledge arrives through a side channel, not the data path.

When `start_load_kv` posts its read, it lands as two NIXL calls: [`makeXferReq`](https://github.com/ai-dynamo/nixl/blob/f4b10ff097daedf8cea7c9e369ea07f4ec1ca3a1/src/api/python/_api.py#L552-L580) builds a transfer from local descriptors and their remote matches, and [`postXferReq`](https://github.com/ai-dynamo/nixl/blob/f4b10ff097daedf8cea7c9e369ea07f4ec1ca3a1/src/api/python/_api.py#L631-L638) launches it. NIXL [scans its configured backends](https://github.com/ai-dynamo/nixl/blob/f4b10ff097daedf8cea7c9e369ea07f4ec1ca3a1/src/api/python/_api.py#L236-L252), sees `UCCL`, and hands the transfer down. The read is no longer NIXL's problem — it's a UCCL operation.

## The UCCL plugin: translating NIXL into UCCL

NIXL doesn't move bytes either — it defines a backend interface and lets plugins implement it. The UCCL plugin is an adapter, and its whole job is translation: NIXL speaks descriptors and opaque metadata blobs, UCCL speaks registered memory and FIFO descriptors, and the plugin converts each NIXL call into the UCCL one beneath it. It [registers under the name `UCCL`](https://github.com/ai-dynamo/nixl/blob/f4b10ff097daedf8cea7c9e369ea07f4ec1ca3a1/src/plugins/uccl/uccl_plugin.cpp#L41-L44) and declares it handles host and GPU memory. When NIXL instantiates it, the backend constructs [`nixlUcclEngine`](https://github.com/ai-dynamo/nixl/blob/f4b10ff097daedf8cea7c9e369ea07f4ec1ca3a1/src/plugins/uccl/uccl_backend.cpp#L84-L97), which spins up a UCCL engine and a [listener thread](https://github.com/ai-dynamo/nixl/blob/f4b10ff097daedf8cea7c9e369ea07f4ec1ca3a1/src/plugins/uccl/uccl_backend.cpp#L141-L165) for incoming connections.

The translation is clearest around UCCL's remote-access primitive, the FIFO. At registration, [`registerMem`](https://github.com/ai-dynamo/nixl/blob/f4b10ff097daedf8cea7c9e369ea07f4ec1ca3a1/src/plugins/uccl/uccl_backend.cpp#L342-L383) registers the local region with UCCL, creates a FIFO for remote access, and serializes it into the public metadata NIXL ships out of band — [`getPublicData`](https://github.com/ai-dynamo/nixl/blob/f4b10ff097daedf8cea7c9e369ea07f4ec1ca3a1/src/plugins/uccl/uccl_backend.cpp#L178-L203). On the peer, [`loadRemoteMD`](https://github.com/ai-dynamo/nixl/blob/f4b10ff097daedf8cea7c9e369ea07f4ec1ca3a1/src/plugins/uccl/uccl_backend.cpp#L434-L491) decodes that serialized FIFO. These are the plugin's half of the metadata exchange NIXL promised a layer up.

When the read is posted, two backend calls fire. [`prepXfer`](https://github.com/ai-dynamo/nixl/blob/f4b10ff097daedf8cea7c9e369ea07f4ec1ca3a1/src/plugins/uccl/uccl_backend.cpp#L503-L585) finds the UCCL connection to the remote agent and patches each remote FIFO item with its real address and size via `uccl_engine_update_fifo` — late-binding coordinates that were only layouts at registration. Then [`postXfer`](https://github.com/ai-dynamo/nixl/blob/f4b10ff097daedf8cea7c9e369ea07f4ec1ca3a1/src/plugins/uccl/uccl_backend.cpp#L592-L707) assembles vectors of local memory-region IDs, addresses, sizes, and remote FIFO items and calls `uccl_engine_read_vector` (or `uccl_engine_write_vector` for a write). That call crosses into the UCCL engine itself.

## The UCCL engine: IPC or the wire

The UCCL engine owns the connection between two peers and the choice of how the bytes actually move. Its first decision is whether the NIC is even needed: if the endpoints share a node — or a process — it can move the data through shared-memory IPC and skip the fabric entirely; otherwise it goes to RDMA over the wire. [`uccl_engine_read_vector`](https://github.com/doublewordai/uccl/blob/c728050453a8d00bd55d50dcbbd72334d8b9948d/p2p/uccl_engine.cc#L319-L362) makes that branch — [local IPC](https://github.com/doublewordai/uccl/blob/c728050453a8d00bd55d50dcbbd72334d8b9948d/p2p/uccl_engine.cc#L350) when the connection is local, else [remote RDMA](https://github.com/doublewordai/uccl/blob/c728050453a8d00bd55d50dcbbd72334d8b9948d/p2p/uccl_engine.cc#L357). A disaggregated transfer across nodes takes the RDMA path.

[`Endpoint::readv_async`](https://github.com/doublewordai/uccl/blob/c728050453a8d00bd55d50dcbbd72334d8b9948d/p2p/engine.cc#L901) is where UCCL's "host-driven" character shows: it does not touch the NIC on the calling thread. It packages the read into a task, enqueues it on a ring buffer, and wakes a proxy thread. The transport runs on the CPU — a proxy thread drains the ring and posts the operations — which is what lets UCCL do its own multi-path splitting and congestion control instead of leaving everything to hardware offload. Completion is polled the same way, through [`uccl_engine_xfer_status`](https://github.com/doublewordai/uccl/blob/c728050453a8d00bd55d50dcbbd72334d8b9948d/p2p/uccl_engine.cc#L478-L481).

## The CXI transport: a host-driven endpoint onto Slingshot

UCCL's transport is pluggable, and `CxiEndpoint` is its Slingshot backend, [selected at startup](https://github.com/doublewordai/uccl/blob/c728050453a8d00bd55d50dcbbd72334d8b9948d/p2p/engine.cc#L250) when `UCCL_P2P_TRANSPORT=cxi`. When the proxy thread drains a read task, it dispatches into this endpoint. It is the last layer that still speaks UCCL's vocabulary; its job is to spend it on libfabric calls.

[`CxiEndpoint::uccl_read_async`](https://github.com/doublewordai/uccl/blob/c728050453a8d00bd55d50dcbbd72334d8b9948d/p2p/cxi/cxi_endpoint.cc#L543) is a thin wrapper over [`post_rma`](https://github.com/doublewordai/uccl/blob/c728050453a8d00bd55d50dcbbd72334d8b9948d/p2p/cxi/cxi_endpoint.cc#L555), where UCCL's descriptor world finally meets libfabric. `post_rma` decodes the remote FIFO metadata, bounds-checks the local and remote ranges against their registered regions, computes the remote offset, and — under a fabric lock — calls [`fi_read`](https://github.com/doublewordai/uccl/blob/c728050453a8d00bd55d50dcbbd72334d8b9948d/p2p/cxi/cxi_endpoint.cc#L601) for a read or [`fi_write`](https://github.com/doublewordai/uccl/blob/c728050453a8d00bd55d50dcbbd72334d8b9948d/p2p/cxi/cxi_endpoint.cc#L604) for a write. The arguments are pure libfabric RMA vocabulary: the local buffer and its MR descriptor, the peer's fabric address, the remote offset, and the remote key. That `fi_read` is where the next layer picks up.

## The CXI provider: where portability ends

libfabric is a portability shim. `fi_read` is the same call whether you're on InfiniBand verbs, sockets, EFA, or Slingshot — that's the whole point of it. The uniformity is paid for in the *provider*: a per-fabric backend that translates the generic request into the hardware's native command vocabulary. On Slingshot that's the `cxi` provider, and `fi_read` reaches it through the endpoint's RMA ops table, where [`.read` is bound to `cxip_rma_read`](https://github.com/ofiwg/libfabric/blob/24d63193433d975a264fa00476e4eb9f853e429f/prov/cxi/src/cxip_rma.c#L972-L976).

From there the read funnels through [`cxip_rma_common`](https://github.com/ofiwg/libfabric/blob/24d63193433d975a264fa00476e4eb9f853e429f/prov/cxi/src/cxip_rma.c#L632), the shared entry point for reads and writes. This is where the abstract "read `len` bytes from remote key `k`" acquires physical coordinates: the target NIC is looked up, the destination fabric address is assembled (`cxi_build_dfa`), and — the detail worth pausing on — the remote key is [turned into a Portals table index](https://github.com/ofiwg/libfabric/blob/24d63193433d975a264fa00476e4eb9f853e429f/prov/cxi/src/cxip_rma.c#L719-L730). Cassini is not an IB-verbs device; its RMA model descends from Portals 4, so a "memory region" on the target is a list entry behind a Portals index, not an rkey on a queue pair. A read also takes the DMA path here, since the inline-data (IDC) path only earns its keep for small writes.

[`cxip_rma_emit_dma`](https://github.com/ofiwg/libfabric/blob/24d63193433d975a264fa00476e4eb9f853e429f/prov/cxi/src/cxip_rma.c#L197) is where the request stops being a libfabric concept. It maps the local landing buffer for the NIC via `cxip_ep_obj_map` — **the GPUDirect seam**, where a region of GPU VRAM is handed an IOVA the NIC can DMA into directly, no bounce through host memory — then fills a `c_full_dma_cmd`: opcode [`C_CMD_GET`](https://github.com/ofiwg/libfabric/blob/24d63193433d975a264fa00476e4eb9f853e429f/prov/cxi/src/cxip_rma.c#L366) (writes set `C_CMD_PUT`), the remote offset, key-derived match bits, and the [local IOVA](https://github.com/ofiwg/libfabric/blob/24d63193433d975a264fa00476e4eb9f853e429f/prov/cxi/src/cxip_rma.c#L386) the reply lands in. That struct *is* the GET, in the form the silicon understands.

## The command queue and the doorbell: the floor

You program a Cassini NIC by writing commands into a ring buffer you share with it, then telling it you've done so. The ring is ordinary memory mapped into both your address space and the NIC's; the telling is a single write to a memory-mapped register the hardware watches. That register is the doorbell — and this ring-plus-doorbell pattern, not `post_send` on a queue pair, is how the whole stack above finally reaches the wire.

[`cxip_txc_emit_dma`](https://github.com/ofiwg/libfabric/blob/24d63193433d975a264fa00476e4eb9f853e429f/prov/cxi/src/cxip_txc.c#L531) sets the traffic class and hands the command down through [`cxip_cmdq_emit_dma`](https://github.com/ofiwg/libfabric/blob/24d63193433d975a264fa00476e4eb9f853e429f/prov/cxi/src/cxip_cmdq.c#L404) to [`cxi_cq_emit_dma`](https://github.com/HewlettPackard/shs-cassini-headers/blob/06f69b003aa362b3137f3e6bcd280f48906eb17c/include/cxi_prov_hw.h#L1281) — the call that crosses out of libfabric into the Cassini command-queue API. `cxi_cq_emit_dma` copies the 64-byte GET into the next ring slot and advances a write pointer that, so far, only the host knows about. Nothing has reached the NIC; the command is just sitting in shared memory. [`cxip_txq_ring`](https://github.com/ofiwg/libfabric/blob/24d63193433d975a264fa00476e4eb9f853e429f/prov/cxi/src/cxip_txc.c#L582-L584) then calls [`cxi_cq_ring`](https://github.com/HewlettPackard/shs-cassini-headers/blob/06f69b003aa362b3137f3e6bcd280f48906eb17c/include/cxi_prov_hw.h#L740), which stores the updated write pointer into the mapped device register — [`*cq->wp_addr = wr_ptr`](https://github.com/HewlettPackard/shs-cassini-headers/blob/06f69b003aa362b3137f3e6bcd280f48906eb17c/include/cxi_prov_hw.h#L751) — and issues a write-combining fence. **That one store is the doorbell.** From the NIC's side, the write pointer just moved; it reads the new command out of the ring and runs the GET.

Why that store reaches hardware and not RAM is decided far earlier, at queue allocation. libcxi's [`cxil_alloc_buf_cmdq`](https://github.com/HewlettPackard/shs-libcxi/blob/a962a9254a00ee6440639b241b3cb05ce38b669d/src/libcxi.c#L957) asks the kernel driver for a queue via [`ioctl(CXI_OP_CQ_ALLOC_BUF)`](https://github.com/HewlettPackard/shs-libcxi/blob/a962a9254a00ee6440639b241b3cb05ce38b669d/src/libcxi.c#L977), then mmaps two things from `/dev/cxi`: the [command ring](https://github.com/HewlettPackard/shs-libcxi/blob/a962a9254a00ee6440639b241b3cb05ce38b669d/src/libcxi.c#L1005-L1008) and the [page holding the write-pointer register](https://github.com/HewlettPackard/shs-libcxi/blob/a962a9254a00ee6440639b241b3cb05ce38b669d/src/libcxi.c#L1015-L1017). [`cxi_cq_init`](https://github.com/HewlettPackard/shs-libcxi/blob/a962a9254a00ee6440639b241b3cb05ce38b669d/src/libcxi.c#L1023) points `wp_addr` at that second mapping. So the innocuous assignment inside `cxi_cq_ring` is a store into BAR-mapped NIC MMIO.

And that is the floor. Below the doorbell there is no more software to follow: the Cassini ASIC takes the command, and the bytes cross the Slingshot fabric — Rosetta switches, link-level congestion control — to the remote NIC, which reads them out of the prefill node's registered KV memory and returns them to land in the IOVA we mapped at the provider. The KV cache has moved.

## Coming back up

A read doesn't block the world while it runs, so completion has to climb back up the same stack to wake the request still parked in `WAITING_FOR_REMOTE_KVS`. The CXI provider reaps a completion event off its event queue; UCCL learns of it through `uccl_engine_xfer_status` polling `Endpoint::poll_async`; NIXL's UCCL backend [`checkXfer`](https://github.com/ai-dynamo/nixl/blob/f4b10ff097daedf8cea7c9e369ea07f4ec1ca3a1/src/plugins/uccl/uccl_backend.cpp#L711-L753) sees the transfer done and fires the UCCL notification; and vLLM, seeing NIXL report completion, [promotes the request out of `WAITING_FOR_REMOTE_KVS`](https://github.com/doublewordai/vllm/blob/e2271c323ed3da1dd3827cc84b142b8b621e61b5/vllm/v1/core/sched/scheduler.py#L2069-L2118) and finally schedules it for decode. The cache is local now; generation can start.

## The whole path

```
vLLM NixlConnector
  -> nixl_agent.makeXferReq / postXferReq
  -> NIXL backend dispatch
  -> nixlUcclEngine::prepXfer / postXfer
  -> uccl_engine_read_vector / write_vector
  -> UCCL Endpoint::readv_async / writev_async   (enqueue, proxy thread posts)
  -> CxiEndpoint::post_rma
  -> libfabric fi_read / fi_write
  -> cxip_rma_common -> cxip_rma_emit_dma         (build c_full_dma_cmd, C_CMD_GET)
  -> cxip_txc_emit_dma -> cxi_cq_emit_dma          (copy command into the ring)
  -> cxi_cq_ring -> *cq->wp_addr = wr_ptr          (the doorbell)
  -> Cassini ASIC executes the GET on the Slingshot wire
```
