The right shape for large scale inference of MoE models is expert parallelism.
Distributed inference requires communications between GPUs. Tensor, context,
fully-sharded, pipeline, parallelisms all require communications between GPUs,
but the comms proceeds in very standard patterns: all the GPUs reach a certain
point, then the same amount of data needs to find its way to the other GPUs in
a very fixed way.

Expert paralelism is different. The routing shape is determined at inference
time. Static comms (an all-reduce followed by an all-gather, as is the fallback
in vLLM) causes redundancy.

Instead, we want to do dynamic, peer-to-peer routing of specific tokens to
specific peer experts, whereever they might be in our network fabric. 

In this post, we'll go through how that works in popular EP kernel libraries

## The job we have to do

Consider data parallel attention (Deepseek, etc.), with each data parallel rank
owning a single GPU. Assume we're working with 8 GPUs, split across 2 nodes,
connected with RDMA. Attention runs on each GPU on a set of tokens $B_i$ that
might vary between GPUs. Assume we're doing expert parallel with $E=8$ experts,
of which $K=2$ are routed for each token. 

At the each rank $r_i$, at the entrance to the EP layer, we have a tensor of
shape $(B_i, H)$ [>1]. The routing layer will run locally, and give us expert
assignments for each token. We're assuming 2-out-of-8 experts: for each token,
the router gives us a set of logits of length $8$ (i.e. a tensor of shape
$(B_i, 8)$), from which we'll get the top-2 argmax, to get a tensor of shape
$(B_i, 2)$. For example, if token $k$ is routed to experts $3$ and $5$, then
row $k$ will be $[3, 5]$. 

So at the entrance to the EP layer each rank holds two things: the activation rows it produced, and, after the local routing pass, the top-2 expert assignment for each of those rows.

Not all of the experts live locally -- some of them live next-door (on
neighbouring NVLink peers), and some of them live far away (available via RDMA
adjacent nodes). The goal of the expert parallelism kernels is to get the
activations where they need to go, run the batched GEMMs when they get there,
and then bring them back home.

We're doing communications here. When we do communications, it can be handy to
specialise based on whether what we care about most is throughput, or latency.
We'll start with the throughput-optimized standard EP shape, then discuss the
latency-optimized shape.

## High throughput

### Dispatch

The point of dispatch is to feed a (batched) GEMM. After routing, every expert has to run
a matrix multiply over exactly the tokens assigned to it. Before routing, those tokens are
scattered across every rank in the cluster. 

So dispatch has to, on each rank, gather the tokens bound for our local experts
into a single dense buffer that a batched GEMM can consume in one go.

The difficulty is that we don't know the shape of that buffer ahead of time.
Which token goes to which expert is decided by the router at runtime, and the
distribution is lumpy and changes every step. An expert might draw a thousand
tokens now and none next time. We can't know how many our local experts will
receive until every other rank has run its router. So neither the size of the
local activation buffer nor the slot in that buffer into which any particular
token lands in is known in advance.

There are two ways to cope with not knowing the size. 

1. We can reserve enough room for the worst case, by allocating a fixed rectangle with a padded slot per expert. 
2. We can find out the real counts first and then allocate exactly what we need. 

The fixed rectangle is simpler, but for throughput it's the wrong trade. It
spends memory on padding, and that memory is far better spent on KV cache,
which is what lets us keep more sequences in flight. When batches are large and
the GEMMs are compute bound, we can afford a little extra communication to
avoid the waste (the opportunity to overlap it with i.e. DBO is greater). So
the throughput path allocates a ragged buffer of the final size,
`[num_local_tokens, H]`, sized to the tokens we'll actually receive. (The fixed
rectangle is the low-latency story, which we'll come to.)

If we want to allocate only what we need, we have to learn the counts before
any activations move. We can do so by running a _coordination pass_. Every rank
already knows from its own routing how many tokens it's sending to each peer.
If everyone trades those numbers, each rank can add up how much it's about
to receive[>2]. 

The coordination pass is cheap, since it's only a handful of integers per peer,
rather than megabytes of hidden state. Once a rank knows how many tokens are
coming from each source, a write-safe layout of the buffer comes naturally as a
prefix sum: the first source's tokens start at zero, the next source's start
where those end, and so on[>3]. The counts hand us both of the things we were
missing, how big to make the buffer and where every block sits inside it.

With the layout fixed, we can actually send the activations. The sender never
writes to the final buffer directly. It streams its tokens into a small
fixed-size queue on the destination, and the receiver, which owns the compact
buffer, drains that queue and copies each token into the slot the prefix sum
assigned it. Going through a queue rather than writing remotely to a computed
address is what lets the two sides run at their own pace, with the depth of the
queue bounding how far ahead the sender can get before it has to wait for the
receiver to catch up.

[>2]: The exchange mirrors the fabric: counts cross between nodes over RDMA, then
between GPUs within a node over NVLink, reducing as they go, so the coordination
costs the same two hops the real dispatch will.

[>3]: The blocks are grouped by source rank, not by expert, because the transfer
is coarser than the routing: a token is sent to a peer if any of its experts live
there, with the per-token expert assignment carried alongside. Regrouping into
the per-expert segments the GEMM wants is a cheap local permutation once the
tokens have arrived.

### Combine

The point of combine is to un-run the dispatch kernel, and add up the
contributions for each token.

The GEMM left its outputs grouped by expert, so the first thing we have to do is to undo the
permutation we did on the way in. The inverse permutation puts the outputs back
into the by-source-rank layout that dispatch delivered tokens in.

From there the transport runs in reverse. The rank that hosted the expert is
now the sender, streaming its outputs back through the the original rank's
per-peer queues back into the positions the tokens originally came from.

So combine mirrors dispatch at every step. The permute becomes an unpermute, the
coordination pass is replaced by reusing dispatch's routing, and where dispatch
compacted arriving tokens into contiguous positions, combine sums them into slots.

[>4]: The transport itself just adds the returning contributions together. The
gate weights are applied separately, either folded into the activations before
the expert GEMM or in the reduction step, so the kernel moving bytes doesn't have
to know about them.

## Low latency


The reason to optimize the kernel for latency is the decode regime. Each rank
holds only a handful of tokens, often one per sequence in the batch. The
coordination pass was cheap in bytes, but it's a full network round trip with
barriers, and it has to finish before any activations move. At decode there is
little to overlap it with, and it becomes a large fraction of the layer [>5].

[>5]: A second important penalty: dynamic shapes like the ones in the
high-throughput kernels are tough to push into CUDA graphs, which are more
important during decode.

So we want to figure out how to skip it. The coordination pass only ever
existed to turn counts into write offsets, and that was only needed to because
the compact buffer made each sender's offset depend on what every other sender
did. If we're willing to give up compactness, we can prearrange space for each
peer rank to write into. Instead of one packed buffer we pre-reserve a fixed,
private region for every (source rank, expert) pair. 

The address a sender writes to is now a formula, the region for its (source,
expert) plus a local slot, and every rank can compute it alone:

$$
\mathrm{addr} = \mathrm{base} + (e \cdot R + r)\,\mathrm{chunk} + \mathrm{slot}
$$

with $e$ the local expert, $r$ the source rank, $R$ the number of ranks, and $\mathrm{chunk}$ the per-region cap. The dynamic
prefix sum over real counts becomes a static stride times a fixed maximum, and
the first thing that happens in the layer is the data send itself.

The catch is that each private region has to be big enough for whatever its
source might send, sized for the worst case rather than the actual count. Left
unbounded that worst case is enormous[>6], so we need to cap how many tokens a
rank may dispatch in one call to a fixed chunk size, and microbatch anything
larger. With the cap each region is one chunk tall, and a receiver's buffer is
$n_\mathrm{local\ experts} \times n_\mathrm{ranks} \times \mathrm{chunk} \times
H$. Each slab is sized for a source dumping its whole chunk into one expert, but
routing spreads those tokens across all the experts, so the slabs sit mostly empty.

Mostly empty means the receiver cannot just hand the buffer to the GEMM,
because most of it is uninitialised. It needs to know how many rows each
source actually wrote. We need to make sure we do that without doing two
passes. When a sender finishes filling its region it writes a single number
into the receiver, the count of tokens it put there, and that number does
double duty: the slot starts empty, so its arrival is also the signal that the
region's data has landed[>7]. The receiver watches the counts, learns the valid
range of every region as it fills, and the grouped GEMM masks the padded rows.

In sum: low latency inverts the throughput tradeoff. Throughput spends a round trip to
keep memory tight; low latency spends memory, in the form of mostly-empty
worst-case buffers, to remove the round trip. 

[>6]: Without a cap, each region would have to hold the most tokens any rank could
ever present, and there is one region per source rank, so the receive buffer would
grow with the total number of tokens in flight across the system rather than with
a fixed budget. The cap replaces that with a constant.

[>7]: It is the same sentinel trick the throughput path used for its host counter.
The count is written in a form the receiver can tell apart from the empty initial
value, so even a source that sends zero tokens produces a signal distinguishable
from one that simply has not arrived yet.

[>1]: $H$ is the hidden size.

## Conclusion

We've gone through how to build the coordination mechanisms & data transfers
for a distributed EP kernel from scratch. The walk through was based on the
[UCCL](https://github.com/uccl-project/uccl) library, which takes the shape of
its expert parallel primitives from the
[DeepEP](https://github.com/deepseek-ai/DeepEP) library.

Doing EP in this way unlocks lots of interesting optimizations. Dynamic expert
load balancing ([EPLB](https://github.com/deepseek-ai/EPLB)), fault tolerance
with dynamic world sizes ([vLLM's elastic EP](https://vllm.ai/blog/2026-05-14-elastic-expert-parallelism)),
and exploiting expert co-activation
([Josh's post](https://blog.doubleword.ai/moe-expert-coactivations)).

Work is ongoing to fuse these kinds of comms primitives into the compute kernels
themselves ([mKernel](https://uccl-project.github.io/posts/mkernel/),
[ParallelKittens](https://hazyresearch.stanford.edu/blog/2025-11-17-pk)), so we
can do fine-grained overlap and better pipelining: one SM can be receiving data,
while the GEMM tiles start firing off of that data.
