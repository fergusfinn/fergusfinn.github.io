## Here's one I made earlier

One way to approach this problem is to take the timeline above and start
hitting it with the optimization hammer. None of this stuff has to take this
much time, and where it does need to take time, it could be cached,
parallelised, pipelined.

It's hard though. SGLang is a big project, with a lot of moving parts. And the
current structure is rational! We should take lots of time in startup to make
sure we're being as performant as we can at runtime. We should expect that they
will stay rational. If we make startup fast, it will later become slow again.

What we really want to do is just take everything that SGLang does on startup
(including whatever they might choose to do in the future) and cache it, all
together in some nice package[^2]. This sort of thing comes under the handle of
'checkpoint/restore' for processes. The leading solution on Linux is the
[CRIU](https://en.wikipedia.org/wiki/CRIU) project, which (with cooperation
from the kernel) can actually checkpoint and restore process state entirely in
user-space. It can checkpoint NVIDIA device state via its in-tree integration
with the [cuda-checkpoint](https://github.com/NVIDIA/cuda-checkpoint) plugin.

Here's how we do when we CRIU checkpoint our SGLang process, and then restore
from our checkpoint:

## Welcome to the machine

32s is good, but still not great. We're spending more time on process
orchestration than we actually are loading weights. Loading weights is slow, if
it's not the bottleneck, something needs fixing.

The first and most visible blob is the `6s` of container setup time. Why did it
go up from when we were launching a regular container? 

The problem turns out to be in containerd.

Most of the container-setup time was going into two needless bits of work that
containerd does on every checkpoint restore, that it doesn't do on normal pod launches. 

For every checkpoint, containerd unpacks the `tarred` checkpoint from disk to a
staging directory, from which it launches the pod. It does this every single
time we launch the pod. This is 6GB of disk writing, that's identical across
restores. For normal container launches, this is skipped if it's been done
before. This saves us a second.

For every checkpoint, containerd tries to resolve the base image from the
remote repository. This happens regardless of the pull policy. This doesn't
happen when we're not restoring a checkpoint! If the image already exists
locally, we don't need to go over the network. This saves us another couple seconds [>2].

- [doublewordai/containerd#1](https://github.com/doublewordai/containerd/pull/1): cache the extracted-checkpoint rootfs so subsequent restores skip the re-unpack.
- [doublewordai/containerd#2](https://github.com/doublewordai/containerd/pull/2): resolve the base image locally instead of unconditionally calling `Pull(config.RootfsImageRef)` inside `CRImportCheckpoint`, even when the base image digest is already present on the node. This eliminated the `1.645s` warm restores were still spending inside `rootfs_image_prepare`.

Together the two patches cut the container-setup phase roughly in half, from `6.0s` to `3.0s`, and tighten run-to-run variance.

## Down where the drunkards roll

We should really, if we're being principled, start hitting that big weight
reload bit. But first, why is CRIU + checkpoint restore going so slowly? 7s to
start a process seems like an awful lot - it's only 6GB of data on disk.
Starting processes is milliseconds, and transferring that data (even cold from
disk, and it's almost certainly in page cache) shouldn't take more than a
second. CRIU makes a lot of syscalls, and does a lot of very cool stuff, but still.

The optimizations break down into two bits:

#### Stop CRIU from copying so much data

- **Zero-copy page restore.** On restore, CRIU allocates new anonymous pages
  and reads into them with `preadv`. The mapping is complex, so there are
  thousands of syscalls, and even when we're hitting page cache we're going
  through so much kernel machinery that you don't see the performance. We've
  got the pages already, in our checkpoint, what we really want is to say to
  the new process - 'look, here's where your pages are, get them when you need
  them'. We can do this by making CRIU `mmap` the pages-img file directly with
  `MAP_PRIVATE|MAP_FIXED`. Pages that are never touched stay as zero-cost
  file-backed references; pages that get written trigger copy-on-write on first
  write.

#### Save ourselves some cuda-checkpoint work

- **Parallel CUDA plugin restore.** SGLang runs three processes with meaningful
  cuda contexts. We can fork the cuda plugin so that `--get-restore-tid`,
  `cuda-checkpoint --action restore`, and `--action unlock` run concurrently
  across GPU-owning processes instead of serially. I'm not sure how much value
  this actually has - across benchmarks it's sometimes meaningful and sometimes
  it isn't - perhaps they serialize on the device side, but sometimes the
  pipelining seems to help.
- **TID cache in the checkpoint.** A meaningful amount of time at restore is
  just spent figuring out how the processes are laid out for restore - work we
  already did on checkpoint. At checkpoint time, patched CRIU writes
  per-process `restore_tid` values into a cache file inside the image. At
  restore time, the plugin loads them instead of calling `cuda-checkpoint
  --get-restore-tid` live. Each live probe costs ~1.1s on this machine, so this
  alone saves about `0.9s`.

## Keep the home fires burning

It's finally time to tackle the big green bit [^3]. We assume we're scaling
from zero - i.e. we don't have a spare GPU running the same model somewhere
else (if we did, then the nice optimization is to load it P2P over NVLink, and
get ourselves 1.8TB/s BW). But! we do have lots of RAM on the device. The best
we can do if we're loading from RAM is PCIe bandwidth - for this machine setup,
64GB/s per GPU. 

How are we going to get there?

The first thing - `/update_weights_from_disk` in SGLang (the call that powers
our weight reload currently) has a harder job than it needs to. It's got to
load from `safetensors` weights, figure out which allocations live where, then
send data into those allocations. Different safetensors shards contain
different parts of the model. 

But we already worked all this stuff out when we
loaded the model the first time. The reason this is what
`update_weights_from_disk` does is it's designed for RL - where the weights on
disk might have changed. `safetensors` is universal, so the training side of RL
can output it, and `update_weights_from_disk` can ingest it.

But we don't care about universality, we just want the model to come up fast.
The [`torch_memory_saver`](https://github.com/fzyzcjy/torch_memory_saver) library
that SGLang uses interposes all of SGLang's cuda allocations, and uses CUDA
virtual memory APIs to keep SGLang's allocation in place while we
`release_memory_occupation`. It supports backing the weights up to CPU when it
does that, but it doesn't support backing them up to disk.

So we add that. We write in a batched device-to-host transfer from torch memory
saver when it is called upon to `release_memory_occupation`. Then on reload,
our job is much simpler - we just need to get the allocations back into their
proper places, from disk. This chops off a little bit of the weight reload from
disk overhead (most importantly - `update_weights_from_disk` is broken for lots
of models. The reason is that it drifts in the code from the original load
path. By making our own checkpoint restore path for the weights, we're not
dependent on this RL path.)

Now - we've got a ~118GB single file that we want to load onto a GPU as fast
as we possibly can. We can add some fast disks, add GPU Direct Storage (GDS),
do all this stuff. But if we don't want to re-spec our machine, and, like I do
here, we've just got an NVMe virtio disk, what can we do? 

We've got to get our weights into RAM before our checkpoint starts restoring.
But the process doesn't exist before it starts restoring. So it's going to have
to get this RAM backed 'weights file' from someone else.

We run a 'daemon' on the device, whose job is to watch the 'weights checkpoint'
directory, and read their contents into RAM buffers. It exposes a unix socket
to the restored containers. Our `torch_memory_saver`, when reloading from disk,
first queries this unix socket. If our daemon has the weights staged, it passes
the FD to the RAM backed ring buffer the weights are staged in over the socket
to the restore process, and reload happens from RAM instead of disk. Otherwise,
we fall back to regular restore. Depending on how much memory you give the
daemon, the chance of a 'cache hit' can be very high or very low.

So the problem becomes - how do you load quickly to device from RAM? If you
just take a RAM buffer and try to H2D transfer it, you get nowhere near PCIe
speeds. The problem is you really need the driver to know about it and to know
that it's in a nice stable location in RAM. Then you can do a 'pinned' host to
device transfer, where loads are issued straight from RAM to the device as DMA.

But this 'registration' process is slow. It scales with the number of `pages`
that back your RAM buffer. We do two things: 
1. `hugepages` - the linux kernel can back a memory buffer by regular `4KiB`
   pages, _or_, if you want, you can use 2MB (or even 1Gi) 'hugepages'. Larger
   pages, mean fewer pages per GB, mean fewer page faults, mean lower
   registration overhead. The daemon owns a pool of hugepages, which it
   allocates into, and hands out over its socket.

2. `pipelining` - I didn't think this would work, but it really does seem to.
   You need to get your buffer registered with the device, and then you need to
   copy the data over to the device. You could call `cudaHostRegister` on the
   whole thing, then `cudaMemcpy` on the whole thing. The advice I've seen is
   that this is rarely worth doing, since the `cudaHostRegister` overhead will
   kill the transfer benefit. This seems not to be the case with hugepages. And also!
   apparently we can do both registration and copying at once (this ought not
   to be too surprising I guess, but I expected them to serialize - otherwise,
   why don't we just do this all the time instead of CPU bounce buffers). So
   you first register a block, then issue an async H2D copy on that block.
   While that copy is executing, you register the next block, &c. &c.

The net effect is big:

## Appendix: How to get SGLang to cooperate

CRIU's integration with cuda-checkpoint goes in stages - the plugin first
calls `cuda-checkpoint` to move the GPU allocations & device state into
host RAM, and then that host RAM gets serialized by CRIU to disk.
Since the B200 we're running this experiment on has 192GB RAM - this means
we get a 192GB checkpoint (actually substantially larger, since we store all
the host memory the process holds too). 

Writing and reading these whole device checkpoints to and from disk takes an
age. And `cuda-checkpoint` by itself gets nowhere near PCIe bandwidth even for
its part of the checkpoint process. 

The problem is that we're pulling a lot of stuff that SGLang already does
really well into our checkpointing process.

1. **KV Cache**: We're checkpointing the entire state of the KV cache. This is
   nice! but it's also completely unnecessary. The KV cache memory hierarchy -
   where we have KV cache pools on Peers, RAM, Disk, Object storage - paged in
   on demand into a running SGLang server, is one of core functionalities of
   modern distributed inference frameworks. So we should just throw away our KV
   cache - if we need it back, there are better places to get it. Once we do
   this, then our checkpoint size scales with the size of the weights, but not
   the size of the GPU.
2. **Weights**: Putting the weights into the checkpoint runs them through a
   process that, again, SGLang is tightly optimized to do better. Just throw
   them away! SGlang knows how to load weights at near NVMe speeds. It can even
   do it via its [RL
    API](https://docs.sglang.io/advanced_features/sglang_for_rl.html) - just
    add `--enable-memory-saver` to the SGLang invocation, then call
    `/release_memory_occupation`, `/resume_memory_occupation` and
    `/update_weights_from_disk`. 

Once we do this, the checkpoint size is only the host memory, the CUDA + torch
state that's left on the GPU when no weights are allocated. The average
checkpoint size is about `6.6GB` - about 30x smaller. 

### Orchestration and packaging

A CRIU checkpoint is a specific set of files on disk. But we want to get back
to where we started. `criu restore` will take those files, and rebuild the
process on the host - we can get a running process that looks exactly like our
original one. 

But our original target wasn't a process running happily unisolated on the
host, it was a container running in kubernetes. So we don't just need to
restart the process. We need to 'reschedule a pod'. This means a couple things:
1. We need to build a packaging format or orchestration system such that we can
   tell Kubernetes that we want to schedule a 'checkpoint restore' - not to
   start a process from scratch. Checkpoint/restore has been moving into
   kubernetes core for a
   [while](https://kubernetes.io/docs/reference/node/kubelet-checkpoint-api/).
   The kubelet even has an HTTP API for creating a checkpoint. I found
   generally that this didn't expose enough control to get this performant
   enough (it tells a lower layer (runc) to checkpoint, but doesn't let you
   configure _how_ it tells it). So instead, we build our checkpointing flow on
   `runc` itself, run from the host on which we make our checkpoint. 

   To create a checkpoint that can be restored by `containerd` (the kubernetes
   container runtime I'm using) requires a specific image format, and an annotation to indicate
   that containerd should restore it using CRIU instead of just launching its
   entrypoint. The result is a standard OCI image - it can be pushed & pulled
   from a repository, run on different nodes, etc. 
2. The restored container, by default, looks exactly like the original one.
   This is obviously kinda the goal - but it has a few interesting
   implications. For one - say you have a machine with 8 GPUs, and you take
   your checkpoint on device 4. When you restore, your checkpointed container
   might get device 2 from the GPU operator. But it will still restore onto
   device 4! All the information on what device it owns gets baked into the
   checkpoint, and all the clever CDI kubernetes stuff gets skipped completely. 

   So you have to build a flow to remap the GPU ids inside the checkpoint. This
   requires support from the driver for 'device migration' - which only landed
   in driver version `580`. And you have to remember in an accessible format in
   the checkpoint image what devices are baked into the cuda context, so you
   can then pass them to the `cuda-checkpoint` tool and remap them to their
   new, CDI allocated GPUs when CRIU runs `cuda-checkpoint restore`. We just
   write the device identities into a container layer (since we have it at
   checkpoint time)
3. Ideally - 'reschedule a pod' means just that - write an kubernetes manifest
   and apply it. You don't want to have to do any extra work to load these
   checkpoints above and beyond scheduling the original pods[>1]. But we just
   said in the last section that we wanted SGLang to cooperate with us - which
   means that we need to call its HTTP API after restore! The way we get around
   this is with a neat trick. We can get the container to 'wake itself up'. The
   way we do it is, when we make the checkpoint, we inject a process (with
   `kubectl exec`) into the container's process tree. This process polls
   constantly for the existence of a specific file, and when that file exists,
   calls `/restore_memory_occupation` (and whatever other post restore hooks we
   might want to call). Then, when CRIU checkpoints the process, this process
   gets quiesced and bundled up with the SGLang process. When the image is
   stopped on disk, we write the file that the process polls for into the OCI
   image. Then - on restore, since the file exists, the 'helper' will
   immediately enter its wake flow!

### Making a checkpoint-able SGLang

Lots of things can make containers and pods hard to checkpoint/restore.
Important ones:
1. Threads that hold a cuda context but don't have anything resident on the
   device. `cuda-checkpoint` ends up choking on them. For SGLang these come
   from `torchinductor`'s compilation process. So we set
   `TORCHINDUCTOR_COMPILE_THREADS=1`, so no spare threads get created.
2. In k8s, lots of stuff in the pod will bind to the pods IP instead of
   localhost. When the checkpoint is restored, the pod will have a different
   IP, and everything dies. The solution is to get everything in SGLang that
   listens on TCP to bind either `0.0.0.0` (i.e. all interfaces) or localhost. 
   For single GPU pods, `GLOO_SOCKET_IFNAME=lo` suffices. MultiGPU pods are a
   whole other kettle of fish that we can go into later.
3. `io_uring` SGLang uses `io_uring` via `uvloop`. `io_uring` can't be
   checkpointed yet. There's an environment variable to force `uvloop` onto
   epoll, but it's (if nothing else needs it) easier to just disable it on the
   host `sudo sysctl -w kernel.io_uring_disabled=2`.
4. `HF_HUB_OFFLINE=1`: Prevents outbound TCP connections to Hugging Face Hub.
  With `tcp-established` in `runc.conf`, any active TCP connection at
  checkpoint time gets preserved. Outbound connections to external hosts would
  fail on restore in a new pod network context, so models should be downloaded
  on the host first. So download all the models first into a mounted dir.

[^3]: - Keep the checkpoint/restore flow, but stop treating weight reload as an unavoidable disk read on wake
    - Run a node-local `torch_memory_saver` daemon that watches the checkpoint directory and stages artifacts into hugepage-backed shared buffers
    - Let restored SGLang processes attach to that staged buffer and restore from host RAM at PCIe speed instead of reloading weights from disk
    - Keep the disk artifact as the source of truth; the daemon is just an accelerator for the wake path
    - This is operationally clean in Kubernetes: a `DaemonSet` owns the hugepage reservation, and restored pods just mount the checkpoint directory plus a Unix socket
    The consumer side is a short ceremony: open a Unix socket, receive the shared memory fd via `SCM_RIGHTS`, `mmap` it in-process, `cudaHostRegister` each `1 GiB` span (pipelined with the H2D DMA), and copy to GPU. `update_weights_from_disk` is never called. The shared-ring drain itself moves `117 GB` at about `38 GB/s` effective, which takes `~3s` end-to-end.
