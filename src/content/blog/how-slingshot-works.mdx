## The problem.

Moving data between machines sounds like a solved problem. TCP/IP works, Ethernet works, every machine has both. But the standard networking stack has a shape that is wrong for certain workloads, and fixing that shape is what the next twenty years of high-performance networking was about.

The standard path for sending data over a socket goes roughly like this: the application hands data to the kernel, the kernel copies it into a socket buffer, the kernel schedules the NIC to transmit, the NIC DMAs from the socket buffer onto the wire. On the receiving end, the NIC DMAs incoming data into a kernel buffer, the kernel copies it into the application's buffer, and notifies the application. There are variations and optimizations, but the basic shape has the kernel involved on both sides and the data touching memory at least twice per hop. For a web server handling HTTP requests this is fine — the per-message overhead is negligible relative to what the application does with the data.

For tightly-coupled distributed computation it is not fine. A fluid dynamics simulation running across a thousand nodes synchronizes every few milliseconds: each rank computes its local update, exchanges boundary data with its neighbors, and waits for everyone to finish before the next step. The time spent waiting is determined by the slowest rank. Software overhead in the networking stack — kernel crossings, memory copies, scheduler latency — adds up directly into wall-clock time and cannot be hidden because the computation cannot proceed until the communication completes. The same problem appears in AI training, where a gradient all-reduce across hundreds of GPUs is a synchronization barrier that every node waits on.

What these workloads need is for data to move directly from an application buffer on one machine to an application buffer on another, without either CPU touching it in the critical path. No kernel crossing on the send side, no kernel crossing on the receive side, no intermediate copies. The receiving CPU should not need to be involved at all — the data should just appear in the right place. This is what remote direct memory access means, and building hardware that provides it reliably and efficiently is the problem InfiniBand set out to solve.

## InfiniBand's answer.

In 1999, the industry agreed there was a problem and sat down together to fix it. Two competing proposals — Future I/O, backed by Compaq, HP, and IBM, and Next Generation I/O, backed by Intel, Microsoft, and Sun — merged into a single effort and produced the InfiniBand Trade Association[>1]. The ambition was extraordinary: InfiniBand was not designed as a networking technology but as a replacement for the entire server I/O stack. PCIe for device I/O, Ethernet for networking, Fibre Channel for storage — all of it, on one fabric.

[>1]: Version 1.0 of the InfiniBand Architecture Specification followed in 2000. The original scope was even broader than the networking story suggests: the IBTA envisaged InfiniBand decomposing server hardware itself — storage, I/O devices, and compute on a single switched fabric, with devices attached as endpoints rather than slots on a local bus. [Wikipedia has a good summary of the formation and early history.](https://en.wikipedia.org/wiki/InfiniBand)

What they built was technically coherent from the ground up. The foundation is credit-based flow control at the link layer: a sender cannot transmit unless the receiver has signaled it has buffer space. This makes the fabric inherently lossless, not as a feature added on top but as a structural consequence of how the protocol works. A buffer cannot overflow because you cannot fill it without permission. That lossless substrate then enables a programming model that would otherwise be impossible: one-sided operations, where a host writes directly into a remote machine's memory, or reads from it, without the remote CPU being involved at all. The receiver's NIC handles it. The programming model that grew up around this — memory registration, queue pairs, completion queues, the verbs API[>2] — is a coherent stack built on the guarantees the fabric provides.

[>2]: "Verbs" is not a formal API specification. The IBTA spec defines a set of abstract operations — `ibv_post_send`, `ibv_open_device`, and so on — that must exist and behave in certain ways, without prescribing an exact interface. The de facto implementation is [libibverbs](https://github.com/linux-rdma/rdma-core), developed by the OpenFabrics Alliance and merged into the Linux kernel in 2005.

InfiniBand lost almost every battle it entered. PCIe won the device I/O bus. Ethernet held general networking. iSCSI and NFS held storage. The only place InfiniBand survived and thrived was high-performance computing, for a specific reason: tightly-coupled scientific simulations — fluid dynamics, molecular dynamics, climate models — run across hundreds or thousands of nodes where every rank synchronizes with every other rank. At that kind of coupling, interconnect latency is a direct ceiling on how fast the simulation runs, and people would pay for a dedicated fabric to push that ceiling up.

That narrowing had a consequence. The founding consortium members mostly lost interest as the empire shrank to a single niche they were not primarily in the business of serving. The company that remained was Mellanox, founded in 1999 specifically to build InfiniBand silicon, which ended up owning the NIC and switch market by default. The people who still needed InfiniBand really needed it, and they had nowhere else to go. That concentration built slowly over twenty years until NVIDIA acquired Mellanox in 2020 for $6.9 billion[>3] on the logic that controlling the GPU and the interconnect together meant owning the full AI infrastructure stack.

[>3]: At the time, NVIDIA's largest acquisition. Intel had made a late attempt to compete with its own fabric, Omni-Path, after acquiring QLogic's InfiniBand assets in 2012 — and then exited the market entirely in 2019, selling the business to Cornelis Networks. That left Mellanox as the uncontested monopolist just as AI training demand was beginning to make InfiniBand switches and NICs very valuable hardware. SemiAnalysis has a good account of what the acquisition meant for the competitive landscape: [Nvidia's InfiniBand Problem](https://newsletter.semianalysis.com/p/nvidias-infiniband-problem-qmx-ai).

The island quality of InfiniBand is inseparable from its technical coherence. It is a complete, purpose-built stack: its own physical layer, its own link-layer flow control, its own routing, its own transport, its own programming model. Each layer was designed against the one below it, and the result works extremely well. The cost is that every machine on an InfiniBand fabric needs InfiniBand cables, InfiniBand switches, InfiniBand NICs, and operators who understand all of it. You cannot partially adopt InfiniBand. Anywhere the benefit was not compelling enough to justify a dedicated fabric, it simply did not go.

## The Ethernet pull.

The cost of InfiniBand's island quality was not just the hardware. It was the operational overhead of running two separate fabrics in parallel. Every server needed two sets of cables, two NICs (one Ethernet for TCP traffic, one InfiniBand for RDMA), two switch fabrics, and teams with expertise in different systems. Storage ran on a third fabric, Fibre Channel, with its own specialists. Large datacenters in the mid-2000s were managing three distinct networks that didn't talk to each other, each with its own cabling, spares, and operational procedures.

The obvious solution was to consolidate everything onto Ethernet, which everyone already owned and understood. The problem was that standard Ethernet is lossy by design: when a switch buffer fills up, it drops packets. For TCP traffic this is acceptable. TCP detects loss and retransmits. For RDMA it is not. A dropped packet in the middle of an RDMA operation causes the queue pair to enter an error state; the application has to tear it down and reconnect. Run RDMA over a lossy fabric and you spend your time recovering from errors rather than moving data.

The fix was Data Center Bridging (DCB), a set of IEEE extensions developed in the late 2000s specifically to make Ethernet lossless enough for storage and RDMA traffic. The key piece is Priority Flow Control (PFC)[>4]: when a switch buffer begins to fill, it sends a PAUSE frame upstream on a per-priority basis, telling the sender to stop transmitting that traffic class before any packets are dropped. Traffic classes that don't need lossless behavior (ordinary TCP) get their own priority lane and are unaffected. Two supporting standards round out the suite: Enhanced Transmission Selection (ETS) schedules bandwidth across priority classes, and DCBX handles discovery and negotiation so that switches and NICs agree on which classes exist and what rules apply to each.

[>4]: PFC is IEEE 802.1Qbb, published 2011; ETS is 802.1Qaz; DCBX is part of 802.1AB. These were developed under the IEEE 802.1 Data Center Bridging task group starting around 2006. The first RoCE specification from the IBTA appeared in 2010, essentially anticipating DCB finalization — the two were developed in parallel, each assuming the other would exist.

PFC achieves losslessness through signaling rather than structure. A PAUSE frame reaches the sender, and if that sender's buffers are also filling, it has to PAUSE its upstream neighbor in turn. The pause can propagate backward through the fabric. In pathological cases (a hot spot, a slow receiver, a misconfigured switch), these cascading pauses can spread congestion across links that had nothing to do with the original problem[>5]. InfiniBand's credit-based flow control prevents this by construction: a sender never fills a buffer it doesn't have credits for, so there is no condition to recover from with emergency signaling. Getting DCB right in a large network is real operational work in a way that InfiniBand's losslessness is not.

[>5]: This failure mode is called a PFC pause storm. Standard mitigations include careful traffic class design, watchdog timers that detect and break deadlocks, and ECN (Explicit Congestion Notification) to signal congestion before buffers fill so that senders back off before a PAUSE is needed. The SemiAnalysis piece on [100,000 H100 clusters](https://newsletter.semianalysis.com/p/100000-h100-clusters-power-network) is good on the operational realities of running RoCE at scale.

But it works well enough. Once you have a lossless Ethernet substrate, the obstacle to running RDMA over it is removed. The question becomes whether you need to rebuild the entire RDMA programming model from scratch, or whether you can port what already exists onto the new substrate. The answer was the latter.

## RoCE: not "RDMA on Ethernet", but literally IB on Ethernet.

The answer was to take the InfiniBand transport layer and run it directly over Ethernet, changing as little as possible. RoCE (RDMA over Converged Ethernet) is not a reimplementation of RDMA semantics for a new substrate. It is InfiniBand transport headers inside Ethernet frames. The Base Transport Header, the queue pair model, the memory registration scheme, the verbs API: unchanged. A NIC that speaks RoCE and a NIC that speaks InfiniBand implement the same programming model; only the link layer beneath them differs.

The first version, RoCE v1, put the IB transport directly inside an Ethernet frame. It was not IP-routable and worked only within a single broadcast domain. RoCE v2 wrapped the same transport in UDP/IP (port 4791), making it routable across subnets and compatible with standard ECMP load balancing across fat-tree topologies[>6]. RoCE v2 is what everyone uses today. The lossless substrate is still required (DCB and PFC configured end to end), but the traffic now looks like UDP to every piece of infrastructure between the endpoints.

[>6]: ECMP (Equal-Cost Multi-Path) hashes flows across parallel paths, which is how a fat-tree distributes traffic. With RoCE v2, a flow is a standard IP 5-tuple and hashes normally. RoCE v1 is Ethernet-only and doesn't hash across IP paths, which limits the topologies it can use.

The economic argument for this at scale is straightforward. An InfiniBand switch is a piece of specialty hardware sold by one company. An Ethernet switch at the same port count and speed is a commodity product with many vendors competing on price. A hyperscaler building a cluster with tens of thousands of GPUs is buying hundreds of switches; the difference in cost per port multiplied across that many ports is substantial. Ethernet also brings the operational advantage: the people who run these networks already know it, already have monitoring and tooling built around it, and don't need a separate team to manage a parallel IB fabric.

This is why the large hyperscalers converged on RoCE for AI training infrastructure. Meta, Microsoft, and Google have all built large GPU clusters on Ethernet with RoCE rather than InfiniBand. At the same link rate the two perform comparably; the choice was about economics and operational simplicity at scale.

The competitive pressure this created is visible in NVIDIA's own product line. Having acquired Mellanox and with it a monopoly on InfiniBand, NVIDIA simultaneously sells Spectrum-X, an Ethernet networking platform optimized for AI workloads, with its own congestion control and adaptive routing layered on top of standard Ethernet[>7]. The IB monopoly did not go unchallenged; the industry moved to a substrate NVIDIA doesn't control, and NVIDIA had to follow.

[>7]: SemiAnalysis covers the competitive dynamics in detail: [Nvidia's InfiniBand Problem](https://newsletter.semianalysis.com/p/nvidias-infiniband-problem-qmx-ai).

## Where things stand.

The market has not settled on a single answer, and probably won't. The three categories that emerged from this history each occupy a different niche.

InfiniBand remains strong in traditional HPC and in NVIDIA's own reference architectures for large GPU clusters. Where performance is the primary constraint and cost is secondary — national supercomputing facilities, or customers buying a complete DGX SuperPOD — the operational simplicity of a purpose-built fabric still wins. NVIDIA sells both the GPU and the interconnect into these environments, and has every incentive to keep IB competitive.

RoCE is winning at hyperscale. The large cloud providers and AI labs building the biggest GPU clusters have converged on Ethernet-based RDMA, driven by economics and the operational leverage of a fabric they already understand. Proprietary fabrics occupy the remaining niches: Slingshot for HPE's government HPC systems[>8], EFA for AWS, which for the same reasons as HPE would rather not depend on NVIDIA for its networking layer at scale[>9].

[>8]: Slingshot descends from Cray's interconnect lineage — Gemini, then Aries — which HPE inherited when it acquired Cray in 2019. The customers are national laboratories and government computing facilities buying complete systems: Frontier at Oak Ridge, Aurora at Argonne, Isambard at Bristol.

What all of these fabrics share is the need for a software abstraction that lets communication libraries target them without being rewritten for each one. That abstraction is libfabric. The next post is a technical map of libfabric on Slingshot: the object model, the available operations, and the ordering and completion semantics — the reference you'd want before writing any code against the fabric.

[>9]: AWS's Elastic Fabric Adapter (EFA) uses their own SRD (Scalable Reliable Datagram) protocol rather than IB or RoCE, exposed through a custom libfabric provider. The motivation is the same as Slingshot's: at AWS's scale, dependency on NVIDIA for the networking layer is unattractive.
