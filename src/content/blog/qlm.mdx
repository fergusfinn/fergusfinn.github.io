A few cloud warehouses now let you call an LLM from a `SELECT` statement. Snowflake calls it [Cortex](https://docs.snowflake.com/en/user-guide/snowflake-cortex/aisql), BigQuery calls it [AI Functions](https://cloud.google.com/blog/products/data-analytics/sql-reimagined-for-the-ai-era-with-bigquery-ai-functions). The proposition is simple: SQL is already the language for transforming tabular data at scale, LLMs are general-purpose text functions, so let the warehouse run the model calls alongside the query[>1].

[>1]: It's a natural fit. Need to classify a million rows of free text? Summarize a column of customer feedback? Extract entities from unstructured data? Write a SQL query, and the warehouse handles the model calls. The thing that makes it economically viable at scale is batched inference: instead of paying realtime API prices per row, you submit prompts in bulk and trade latency for throughput.

The same idea works outside a warehouse. [qlm](https://github.com/doublewordai/qlm) is a small SQL shell built on [DataFusion](https://datafusion.apache.org/) that adds three LLM functions to standard SQL — `llm()`, `llm_unfold()`, `llm_fold()` — and routes the calls through Doubleword's [Batched](https://docs.doubleword.ai/batches) inference service[>2]. No warehouse, no server, no database extension. Point it at a local parquet file, write a query, get results. The [README](https://github.com/doublewordai/qlm) has the full API. This post is about what the primitives let you do.

[>2]: Batch inference is what makes this practical for ad-hoc local analysis. Calling an LLM per row across tens of thousands of rows becomes a normal thing to do, not a budget event.

The dataset is the [U.S. Congressional Speeches corpus](https://huggingface.co/datasets/Eugleo/us-congressional-speeches): 17 million speeches from 1873 to 2010, split across 28 parquet shards. Pointing qlm at a directory loads all the shards as a single table:

```bash
mkdir -p speeches && cd speeches
for i in $(seq 0 27); do
  wget -q "https://huggingface.co/datasets/Eugleo/us-congressional-speeches\
/resolve/main/data/train-$(printf '%05d' $i)-of-00028.parquet"
done
cd ..
qlm --api-key $DOUBLEWORD_API_KEY -t speeches=speeches/
```

## What was Congress arguing about?

The simplest primitive is `llm()`. It takes a prompt template and one or more columns, substitutes the column values into the template, and calls the model once per row — a `MAP` operation where the function is "ask the model". The `{0}`, `{1}` placeholders refer to column arguments by position.

To get a feel for the dataset, sample three speeches per decade and ask the model what each one is about:

```sql
SELECT decade, topic FROM (
  SELECT decade,
    llm('What specific topic is debated in this speech? '
        'Answer in under 15 words. Be concrete — name '
        'people, bills, places, or dollar amounts:\n\n{0}',
    text) as topic,
    ROW_NUMBER() OVER (
      PARTITION BY decade ORDER BY RANDOM()) as rn
  FROM (
    SELECT text, CONCAT(LEFT(date, 3), '0s') as decade
    FROM speeches
    WHERE char_count > 1000 AND char_count < 4000
  )
)
WHERE rn <= 3 ORDER BY decade, rn;
```

## A bird's-eye view

That's 42 LLM calls. With batched inference cheap enough to make this routine, we can scale up: classify ~5,500 speeches (40 per year across every year from 1873 to 2010) into broad topic categories, and plot how Congressional attention has shifted over 140 years.

```sql
SELECT LEFT(date, 4) as year, topic, COUNT(*) as count
FROM (
  SELECT date,
    llm('Classify this Congressional speech into exactly one topic: '
        'economy, defense, civil-rights, healthcare, education, '
        'immigration, environment, judiciary, infrastructure, '
        'foreign-policy. One hyphenated word only:\n\n{0}',
    text) as topic
  FROM (
    SELECT text, date,
      ROW_NUMBER() OVER (
        PARTITION BY LEFT(date, 4) ORDER BY RANDOM()) as rn
    FROM speeches
    WHERE char_count > 500 AND char_count < 5000
  )
  WHERE rn <= 40
)
GROUP BY year, topic ORDER BY year, count DESC;
```

The trends tell a clear story. Judiciary dominates the 19th century — Congress was building the legal infrastructure of a new nation. Economy spikes in the 1930s (the Depression) and again post-2008. Defense surges during WWII, foreign policy peaks during the Cold War. Civil rights rises through the 1960s and reaches its highest share in the 1990s. Healthcare barely registers before 1950 but climbs steadily to 13% by the 2000s. Environment appears as a topic only after 1970 — the year of the EPA and the Clean Air Act.

## Who is Congress talking about?

`llm_unfold()` is for one-to-many transformations: one input row, a variable number of output values. The model returns a delimited list, qlm splits it into an array, and `UNNEST()` expands it back into rows. To extract every foreign country mentioned in a speech and aggregate by year:

```sql
SELECT LEFT(date, 4) as year, TRIM(country) as country, COUNT(*) as mentions
FROM (
  SELECT date,
    UNNEST(llm_unfold(
      'List the foreign countries discussed in this speech. '
      'Comma-separated, country names only. If none, reply "none":\n\n{0}',
      text, ',')) as country
  FROM (
    SELECT text, date,
      ROW_NUMBER() OVER (
        PARTITION BY LEFT(date, 4) ORDER BY RANDOM()) as rn
    FROM speeches
    WHERE char_count > 500 AND char_count < 5000
  )
  WHERE rn <= 40
)
WHERE TRIM(country) != 'none' AND TRIM(country) != ''
GROUP BY year, TRIM(country) ORDER BY year, mentions DESC;
```

Another ~5,500 LLM calls, but now each call returns a list rather than a single value.

## What did each decade *feel* like?

The topic chart and country sparklines are quantitative views. For a textual companion, `llm_fold()` is an aggregate function — it reduces many rows into one value, like `SUM` or `COUNT`, but using LLM calls. Given N rows, it pairs them up, reduces each pair, then pairs the results, recursively, until one value remains. That's O(log N) LLM calls instead of O(N), which matters once N gets into the thousands.

The classic version of tree-reduce uses an optional map step to compress each row before the reduce, so the reduce prompt only ever sees small summaries. Modern context windows make that mostly unnecessary — at 50 speeches per decade and ~3K characters per speech, even a leaf-level 2-way reduce is only seeing ~6K characters. Skip the map step entirely; the reduce prompt just needs to be neutral about whether its inputs are raw speeches (at the leaves) or prior summaries (at higher levels).

```sql
SELECT decade,
  llm_fold(
    text,
    'Combine the following Congressional material into a 3-sentence '
    'overview of the decade. Focus on dominant political themes, '
    'recurring conflicts, and what was at stake. Be concrete — name '
    'people, bills, dollar amounts where they appear:\n\n'
    'Material A:\n{0}\n\n---\n\nMaterial B:\n{1}'
  ) as summary
FROM (
  SELECT text, CONCAT(LEFT(date, 3), '0s') as decade,
    ROW_NUMBER() OVER (
      PARTITION BY CONCAT(LEFT(date, 3), '0s') ORDER BY RANDOM()) as rn
  FROM speeches
  WHERE char_count > 500 AND char_count < 5000
)
WHERE rn <= 50
GROUP BY decade ORDER BY decade;
```

## Can the model date a speech from its content?

Here's a different kind of question. Can the model tell *when* a speech was given, purely from the text? No dates, no metadata, just the words.

```sql
SELECT LEFT(date, 4) as actual_year,
  CONCAT(LEFT(date, 3), '0s') as actual_decade,
  llm('What decade was this U.S. Congressional speech from? '
      'Answer with just the decade (e.g. 1950s):\n\n{0}',
  LEFT(text, 2000)) as guessed_decade
FROM (
  SELECT text, date,
    ROW_NUMBER() OVER (
      PARTITION BY LEFT(date, 4) ORDER BY RANDOM()) as rn
  FROM speeches
  WHERE char_count > 500 AND char_count < 5000
)
WHERE rn <= 40
ORDER BY actual_year;
```

## Closing

Each of these queries is doing something that, ten years ago, would have been a Python script with an API client, rate limiting, retry logic, a results pipeline, and a serialization format. Now they're SQL. The warehouses got there first, but you don't need a warehouse: a parquet file, a binary, and a batch API are enough.

qlm is on [GitHub](https://github.com/doublewordai/qlm) (`cargo install qlm`). The README has the full function reference, including semantic search via `vector_search()`, which didn't make it into this post.
